# HabitCircle 핵심 기능 우선순위 전문가 토론 보고서

**일시:** 2026-03-27
**목적:** 서비스 성공에 직결되는 기능을 선별하고, 구현 순서를 확정
**참여 전문가 역할:** 리텐션 엔지니어, 습관 형성 심리학자, 소셜 프로덕트 디자이너, 모바일 그로스 전략가, 데이터 분석가

---

## 전제: HabitCircle의 현재 위치

### 잘 되어 있는 것 (건드리지 않을 것)
- 원터치 습관 체크 + 즉각적 시각 피드백
- 마일스톤 축하 시스템 (7/14/21/30/50/60/90/100/365일)
- 정체성 기반 온보딩 ("나는 건강한 사람이다")
- 손실 회피 경고 ("연속 기록을 지켜주세요!")
- 소셜 프루프 넛지 ("친구가 오늘 습관을 완료했어요")

### 치명적으로 빠진 것 (이번 토론의 주제)
1. 알림/리마인더가 전혀 없음
2. 하루만 빠져도 스트릭 완전 리셋
3. 매일만 가능, 주3회 같은 유연한 빈도 불가
4. 친구에게 "잘했어!" 한마디를 보낼 수 없음
5. 주간/월간 리뷰 없음

## 논제 1: "앱을 안 열면 습관도 안 한다" — 알림

### 리텐션 엔지니어:

> 습관 앱에서 알림이 없는 건 알람 없는 시계와 같습니다.

**데이터 근거:**
- Duolingo: 푸시 알림 도입 후 DAU +25%, D7 리텐션 +12%
- Headspace: 명상 리마인더가 7일 연속 사용률을 2배로 끌어올림
- Noom: 개인화된 알림 시간이 일반 알림 대비 클릭률 3.4배

**HabitCircle 현재 상태:**
- 알림 인프라: 완전히 0. 패키지, 서비스워커, DB 테이블 모두 없음
- 사용자가 앱을 열지 않으면 스트릭이 깨지는데, 앱을 열라는 신호가 없음

### 습관 형성 심리학자:

> 습관 루프는 **신호(Cue) → 루틴(Routine) → 보상(Reward)**입니다. HabitCircle은 루틴과 보상은 있지만, **신호가 없습니다.**

**알림이 해야 할 역할:**

| 시점 | 알림 내용 | 심리 원리 |
|------|----------|----------|
| 아침 (사용자 설정) | "좋은 아침! 오늘의 습관을 시작해볼까요?" | 신호(Cue) 제공 |
| 저녁 8시 (미완료 시) | "📚 독서, 💧 물 2L 아직 남았어요" | 손실 회피 |
| 스트릭 위험 (밤 10시) | "🔥 14일 연속 기록이 오늘 끊길 수 있어요!" | 긴급성 |
| 친구 활동 | "민수님이 방금 운동을 완료했어요 💪" | 소셜 프루프 |
| 마일스톤 달성 | "🎉 30일 연속 달성! 대단해요!" | 보상 강화 |

### 모바일 그로스 전략가:

**구현 방식 결정:**

| 방식 | 장점 | 단점 | 권장 |
|------|------|------|------|
| Web Push (브라우저) | 네이티브 앱 없이 가능 | iOS Safari 지원 제한적, 허용률 낮음 | 1순위 |
| 이메일 리마인더 | 100% 도달률, 구현 간단 | 실시간성 떨어짐, 스팸 처리 위험 | 2순위 (보조) |
| 카카오 알림톡 | 한국 시장 최적, 열람률 최고 | 유료 (건당 ~8원), API 연동 필요 | 3순위 (유료 후) |

**합의:** Web Push를 1순위로, 이메일 다이제스트를 보조로 구현.

### 구현 설계

**필요한 것:**
```
1. public/sw.js — 서비스 워커 (푸시 수신)
2. public/manifest.json — PWA 매니페스트
3. DB: notification_preferences 테이블
   - user_id, reminder_time, enabled, push_subscription
4. DB: notification_log 테이블
   - user_id, type, sent_at
5. API: /api/notifications/subscribe (구독 등록)
6. API: /api/notifications/send (서버 발송)
7. 스케줄러: 저녁 미완료 체크 → 알림 발송
```

**난이도:** 높음 (3-4일)
**임팩트:** 최상 (D30 리텐션 +15~25% 예상)

## 논제 2: "하루 빠졌다고 30일이 사라진다" — 스트릭 프리즈

### 습관 형성 심리학자:

> 완벽주의는 습관 형성의 적입니다. "한 번 실패 = 전부 실패"라는 인식이 들면 사람은 아예 포기합니다. 이것을 **'What-the-hell effect'**라고 합니다.

**현재 HabitCircle의 스트릭 계산 (`streaks.ts`):**
```
29일 연속 완료 → 30일째 하루 빠짐 → 스트릭: 0
```

이건 사용자를 벌주는 것이지, 돕는 것이 아닙니다.

### 리텐션 엔지니어:

**Duolingo의 교훈:**
- 스트릭 프리즈 도입 후 이탈률 -17%
- 무료 프리즈 1개/주 + 유료 추가 프리즈 = 수익 모델에도 기여
- "프리즈 사용됨" 알림이 다음날 복귀율을 +40% 높임

### 전문가 합의 — 스트릭 프리즈 설계:

**무료:** 주 1회 자동 프리즈
**프리미엄:** 주 3회 프리즈

**작동 방식:**
1. 자정(KST) 기준으로 어제 체크 안 한 습관 확인
2. 프리즈 남아있으면 자동 사용 → 스트릭 유지
3. 프리즈 없으면 → 스트릭 리셋 (현재와 동일)
4. "프리즈가 사용되었어요! 오늘은 직접 체크해주세요" 알림

**구현:**
```
1. users 테이블에 streak_freezes_remaining (기본값 1) 추가
2. habit_logs에 was_frozen 컬럼 추가
3. streaks.ts 수정: frozen 로그를 연속일로 카운트
4. 매주 월요일 자정에 프리즈 카운트 리셋 (스케줄러 또는 로그인 시 체크)
```

**난이도:** 낮음 (1일)
**임팩트:** 상 (이탈 방지의 핵심)

## 논제 3: "매일이 부담이에요" — 습관 빈도 설정

### 습관 형성 심리학자:

> James Clear는 "2분 규칙"을 말합니다. 습관은 작게 시작해야 합니다. 하지만 **매일**이라는 빈도 자체가 많은 사람에게 부담입니다.

**실제 사용자 시나리오:**
- 운동: 주 3-4회가 현실적 (매일 운동은 오히려 해로움)
- 독서: 주 5회 (주말 제외)
- 요가: 주 2-3회
- 블로그 글쓰기: 주 1회

**현재 HabitCircle:**
- DB 스키마에 `frequency TEXT DEFAULT 'daily'`가 있지만 **어디서도 사용하지 않음**
- 모든 습관이 무조건 매일로 처리됨
- 주 3회 운동 습관을 만들면, 쉬는 날에도 미완료로 카운트 → 스트릭 깨짐 → 이탈

### 데이터 분석가:

**빈도 지원 시 스트릭 계산 변경:**

| 빈도 | 스트릭 규칙 | 예시 |
|------|-----------|------|
| 매일 | 연속 일수 (현재와 동일) | 7일 연속 = 7 |
| 주 N회 | 연속 주 수 (해당 주에 N회 이상 완료) | 3주 연속 주 3회 = 3주 |
| 평일만 | 연속 평일 수 (주말 자동 스킵) | 10 평일 연속 = 10 |

### 전문가 합의 — 빈도 설계:

**지원할 빈도 옵션:**
1. 매일 (기본값, 현재와 동일)
2. 평일만 (월-금)
3. 주 N회 (1-6, 요일 미지정 — 유연)
4. 특정 요일 (월/수/금 같은 지정)

**구현:**
```
1. habits 테이블의 frequency 컬럼 활용
   - 'daily' | 'weekdays' | 'weekly:3' | 'days:mon,wed,fri'
2. 습관 생성 UI에 빈도 선택 추가 (habits/new/page.tsx)
3. streaks.ts 수정: 빈도에 따른 "예상 완료일" 계산
4. dashboard: 오늘 해당되지 않는 습관은 "오늘은 쉬는 날" 표시
5. log API: 빈도에 해당하지 않는 날에도 체크 허용 (보너스 완료)
```

**난이도:** 중간 (2일)
**임팩트:** 상 (사용자층 확대 + 이탈 방지)

## 논제 4: "잘했어! 한마디가 없다" — 응원 보내기

### 소셜 프로덕트 디자이너:

> Instagram의 좋아요, Strava의 Kudos, Duolingo의 친구 응원 — 모두 **최소한의 상호작용으로 감정적 연결**을 만듭니다.

**현재 HabitCircle의 소셜:**
- 친구 피드: "민수님이 운동을 완료했어요" → 읽기만 가능
- 반응 수단: 0
- 결과: 정보 게시판이지, 소셜 네트워크가 아님

### 습관 형성 심리학자:

> 습관 형성에서 **사회적 강화(Social Reinforcement)**는 자기 보상보다 1.5배 강력합니다. "누군가 내 노력을 알아준다"는 느낌이 지속의 핵심입니다.

### 전문가 합의 — 응원 시스템 설계:

**의도적으로 단순하게:**

```
피드 아이템 옆에 [👏 응원] 버튼 하나.
누르면:
1. 상대에게 "홍길동님이 응원을 보냈어요! 👏" 알림
2. 피드 아이템에 "👏 2" 카운트 표시
3. 하루에 같은 사람에게 1번만 가능 (스팸 방지)
```

**왜 이모지 리액션이 아니라 응원 하나인가:**
- 선택지가 많으면 → 아무것도 안 누름 (선택 과부하)
- 하나의 명확한 액션 → 누르는 비율 3배 높음 (Strava Kudos 데이터)

**구현:**
```
1. DB: cheers 테이블
   - id, from_user_id, to_user_id, habit_log_id, created_at
   - UNIQUE(from_user_id, habit_log_id) — 중복 방지
2. API: POST /api/feed/cheer
   - { habitLogId } → 응원 생성 + 알림 트리거
3. API: GET /api/feed 수정
   - 각 피드 아이템에 cheer_count, has_cheered 추가
4. UI: 피드 아이템에 👏 버튼 + 카운트
5. 대시보드: "오늘 3명이 응원을 보냈어요" 배너 (선택)
```

**난이도:** 낮음 (1일)
**임팩트:** 중상 (감정적 연결 = 장기 리텐션)

## 논제 5: "이번 주 얼마나 했지?" — 주간 리뷰

### 데이터 분석가:

> 측정하지 않으면 개선할 수 없습니다. 사용자가 자신의 패턴을 보지 못하면 행동 변화가 일어나지 않습니다.

**현재 통계:**
- 프로필: 총 완료 수, 최고 스트릭, 습관별 7일 완료율
- 습관 상세: 35일 히트맵, 7일/30일 완료율

**빠진 것:**
- "이번 주 vs 지난주" 비교
- 요일별 패턴 ("수요일에 가장 많이 빠뜨림")
- 주간 트렌드 ("3주 연속 완료율 상승 중!")
- 습관별 "건강 점수" (종합 지표)

### 소셜 프로덕트 디자이너:

**주간 리뷰의 진짜 가치는 "공유 가능한 성취":**
- 주간 리포트 카드 → 카카오톡/Instagram 공유
- "이번 주 87% 달성, 3주 연속 상승 중!" → 자랑하고 싶게 만듦
- 이것이 **무료 광고**가 됨

### 전문가 합의 — 주간 리뷰 설계:

**주간 리뷰 화면 구성:**

```
┌─────────────────────────────────┐
│  📊 이번 주 리뷰 (3/21 - 3/27)  │
│                                 │
│  완료율: 87% (↑ 지난주 72%)      │
│  ████████████████░░░            │
│                                 │
│  ── 습관별 ──                    │
│  📚 독서   ███████ 100% (7/7)   │
│  🏃 운동   █████░░  71% (5/7)   │
│  💧 물     ██████░  86% (6/7)   │
│                                 │
│  ── 인사이트 ──                   │
│  💡 수요일에 가장 많이 빠뜨려요    │
│  🔥 독서가 3주 연속 100%!        │
│  ⚡ 전체 완료율이 3주째 상승 중    │
│                                 │
│  [카드로 공유하기]  [닫기]         │
└─────────────────────────────────┘
```

**구현:**
```
1. API: GET /api/stats/weekly?week=2026-W13
   - 이번 주 + 지난주 완료율
   - 습관별 주간 완료율
   - 요일별 완료 패턴
2. UI: /dashboard 또는 별도 /review 페이지
   - 매주 월요일에 지난주 리뷰 자동 표시
   - 공유 카드 생성 (Canvas API)
3. 알림: "지난주 리뷰가 준비됐어요!" (월요일 오전)
```

**난이도:** 중간 (2일)
**임팩트:** 중 (성취감 + 바이럴)

## 최종 합의: 구현 우선순위

### Tier 1: "이것 없이는 서비스가 안 된다" (이번 스프린트)

| 순서 | 기능 | 핵심 이유 | 난이도 | 소요 |
|------|------|----------|--------|------|
| **1** | **스트릭 프리즈** | 이탈 방지의 즉각적 효과, 구현 가장 간단 | 낮음 | 1일 |
| **2** | **응원 보내기** | 소셜 연결의 핵심, 가장 적은 노력으로 큰 효과 | 낮음 | 1일 |
| **3** | **습관 빈도 설정** | "매일"의 부담 해소, 사용자층 확대 | 중간 | 2일 |

### Tier 2: "이것이 성장을 만든다" (다음 스프린트)

| 순서 | 기능 | 핵심 이유 | 난이도 | 소요 |
|------|------|----------|--------|------|
| **4** | **알림/리마인더** | 앱 복귀 유도, 습관 루프의 "신호" | 높음 | 3-4일 |
| **5** | **주간 리뷰** | 성취 시각화 + 공유 = 바이럴 | 중간 | 2일 |

### 왜 이 순서인가 (전문가 합의 근거)

**스트릭 프리즈가 1순위인 이유:**
- 알림보다 먼저 해야 하는 이유: 알림을 보내도 **하루 빠지면 30일이 리셋**되면, 사용자는 "알림 받아도 의미 없다"고 느낌
- 프리즈가 있어야 "어제 못했지만 스트릭은 살아있다 → 오늘은 해야지" 복귀 동기가 생김
- 구현 비용이 가장 낮음 (streaks.ts + schema 변경)

**응원이 2순위인 이유:**
- 알림 시스템 없이도 즉시 효과 발휘 (앱 내 소셜 강화)
- 구현 비용 매우 낮음 (1개 API + UI 버튼 하나)
- "누군가 나를 보고 있다"는 느낌 → 내일도 체크하게 만듦

**알림이 Tier 2로 간 이유:**
- 가장 임팩트가 크지만 구현 비용도 가장 큼 (서비스워커, VAPID, DB, 스케줄러)
- 스트릭 프리즈 + 응원이 먼저 있어야 알림의 효과가 극대화됨
  - 프리즈 없이 "스트릭이 끊길 수 있어요!" 알림 → 이미 끊겼으면 무의미
  - 응원 없이 "친구가 완료했어요" 알림 → 반응할 수단이 없으면 공허

## 구현 상세 스펙

### 1. 스트릭 프리즈

**DB 변경:**
```sql
ALTER TABLE users ADD COLUMN streak_freezes_remaining INTEGER DEFAULT 1;
ALTER TABLE users ADD COLUMN streak_freezes_reset_at TEXT;
ALTER TABLE habit_logs ADD COLUMN was_frozen INTEGER DEFAULT 0;
```

**streaks.ts 수정:**
- `was_frozen = 1`인 로그를 "완료"로 간주하여 스트릭 연속 카운트
- 프리즈 로그는 히트맵에서 다른 색상으로 표시 (예: 파란색 ❄️)

**자동 프리즈 적용 로직 (로그인 시 또는 자정 체크):**
```
1. 어제 날짜의 로그가 없는 습관 확인
2. freezes_remaining > 0이면:
   - habit_logs에 (habit_id, user_id, yesterday, was_frozen=1) INSERT
   - freezes_remaining -= 1
3. freezes_remaining == 0이면:
   - 스트릭 리셋 (현재와 동일)
4. 매주 월요일: freezes_remaining을 1로 리셋
```

### 2. 응원 보내기

**DB 변경:**
```sql
CREATE TABLE IF NOT EXISTS cheers (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  from_user_id INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  to_user_id INTEGER NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  habit_log_id INTEGER NOT NULL REFERENCES habit_logs(id) ON DELETE CASCADE,
  created_at TEXT DEFAULT (datetime('now')),
  UNIQUE(from_user_id, habit_log_id)
);
```

**API:**
- `POST /api/feed/cheer` — `{ habitLogId }` → 응원 생성
- `GET /api/feed` 수정 — 각 아이템에 `cheerCount`, `hasCheered` 추가

**UI:**
- 피드 아이템에 `👏` 버튼 + 카운트
- 눌렀을 때: 낙관적 업데이트 + 작은 파티클 애니메이션
- 대시보드 상단: "오늘 N명이 응원을 보냈어요" (cheerCount > 0일 때)

### 3. 습관 빈도 설정

**frequency 포맷:**
```
'daily'           — 매일 (기본값)
'weekdays'        — 월-금
'weekly:3'        — 주 3회 (요일 미지정)
'days:mon,wed,fri' — 특정 요일
```

**수정 파일:**
- `habits/new/page.tsx` — 빈도 선택 UI 추가
- `api/habits/route.ts` — POST에 frequency 파라미터 수용
- `streaks.ts` — 빈도별 스트릭 계산 로직
- `dashboard/page.tsx` — 오늘 해당 안 되는 습관은 "쉬는 날" 표시
- `api/habits/[id]/log/route.ts` — 비해당일에도 체크 허용 (보너스)
